# フェーズ別ユーザー判断事項

> 最終更新: 2026-03-05

各フェーズを進める上で、ユーザーの判断・意思決定が必要な項目をまとめる。
技術的に一意に決まるものは除外し、**トレードオフがある選択肢**に絞っている。

---

## 凡例

| 記号 | 意味 |
|------|------|
| **[必須]** | フェーズ着手前に決定が必要（ブロッカー） |
| **[推奨]** | 早期に決めると設計がスムーズ |
| **[任意]** | フェーズ途中〜完了時に決めれば十分 |

---

## Phase 1: Excel パーサー + プレビュー

### 1-1. Excel パースライブラリの選定 [必須]

`.xlsx` ファイルの読み取りに使用するライブラリ。ドメイン層の `excel/` 集約の実装基盤となるため、Phase 1 着手前に決定が必要。

| 観点 | ExcelJS | SheetJS (xlsx) |
|------|---------|----------------|
| ライセンス | MIT | Apache-2.0 |
| ページマージン取得 | ワークシートの `pageSetup` から取得可能 | `!margins` プロパティで取得可能 |
| 印刷スケーリング | `pageSetup.scale` / `fitToPage` 取得可能 | `!print` オブジェクトで取得可能 |
| セル結合情報 | `worksheet.model.merges` | `!merges` |
| バンドルサイズ | 約 1.5 MB | 約 800 KB（mini 版あり） |
| ブラウザ対応 | 対応（ただしNode向け設計） | ブラウザファースト設計 |
| 型定義 | 同梱 | `@types/xlsx` で提供 |
| 印刷エリア検出 | `worksheet.pageSetup.printArea` | `!ref` + `!print` |

**判断基準の例:**
- ブラウザバンドルサイズを重視 → SheetJS
- API の明快さ・型安全性を重視 → ExcelJS
- 両方を PoC で実際に試してから決定する方法もある

**推奨:** ADR-002 として選定理由を記録する。

---

### 1-2. プレビューのズーム・パン操作 [推奨]

SVG プレビューキャンバスでのユーザー操作範囲。

| 選択肢 | 説明 | 工数 |
|--------|------|------|
| **A. Fit-to-window のみ** | ウィンドウに収まるよう自動スケール | 小 |
| **B. ズームのみ** | 50%〜200% 程度のズーム操作を提供 | 中 |
| **C. ズーム + パン** | ズームに加え、ドラッグによるスクロール | 大 |

**推奨:** Phase 1 では **A（Fit-to-window）** で着手し、Phase 2 以降で B/C に拡張する。

---

### 1-3. 複数ページの取り扱い [推奨]

Excel の fitToPage 設定で複数ページに分割される場合の対応方針。

| 選択肢 | 説明 |
|--------|------|
| **A. 1ページ目のみ** | v1 のスコープを縮小し、単一ページに集中 |
| **B. ページ選択式** | ユーザーが変換対象ページを選択する |
| **C. 全ページ一括** | 全ページを変換してタブ切替で表示 |

**推奨:** **A（1ページ目のみ）** で v1 を完成させ、複数ページは後続フェーズで対応する。

## Phase 2: 調整エディタ

### 2-1. 状態管理ライブラリの選定 [必須]

Phase 2 では複数コンポーネント間の状態共有（ボックス選択、プロパティパネル同期、Undo/Redo）が必要になる。React hooks のみでは複雑になるため、ライブラリ導入を検討する。

| 観点 | Zustand | Jotai |
|------|---------|-------|
| パラダイム | 単一ストア（Flux 系） | アトミック（Recoil 系） |
| バンドルサイズ | 約 1.1 KB (gzip) | 約 2.4 KB (gzip) |
| 学習コスト | 低い（シンプルな API） | 中程度（atom 設計の理解が必要） |
| Undo/Redo との相性 | ミドルウェアで対応可 | 派生 atom で対応可 |
| DevTools | 公式対応 | 公式対応 |

**判断基準の例:**
- シンプルさ・チーム馴染みやすさ重視 → Zustand
- 細粒度のリアクティビティ重視 → Jotai

**推奨:** Phase 1 完了後、エディタの状態設計と合わせて決定する。

---

### 2-2. ドラッグ操作の実装方式 [必須]

ボックスの移動・リサイズに使用する方式。

| 選択肢 | 説明 | メリット | デメリット |
|--------|------|----------|-----------|
| **A. ネイティブ SVG イベント** | SVG 要素の mousedown/mousemove で直接実装 | 依存なし、軽量 | 実装量が多い |
| **B. react-dnd** | React 用 D&D ライブラリ | エコシステム成熟 | SVG との相性に注意 |
| **C. @dnd-kit/core** | モダンな D&D ライブラリ | アクセシビリティ対応 | SVG カスタマイズ必要 |

**推奨:** SVG ベースのエディタでは **A（ネイティブ SVG イベント）** が最も制御しやすい。ライブラリは HTML DOM 前提のものが多く、SVG との統合に追加工数がかかる場合がある。

---

### 2-3. スナップ・グリッドの粒度 [推奨]

ボックス配置時のスナップ動作の設定。

| 項目 | 選択肢 |
|------|--------|
| グリッド単位 | 0.5mm / 1mm / 2mm / ユーザー設定可 |
| スナップ距離 | 2px / 5px / 自動（ズーム連動） |
| ガイド線表示 | 常時 / ドラッグ中のみ / 非表示 |

**推奨:** グリッド 1mm、ガイド線はドラッグ中のみ表示を初期値とし、設定パネルで変更可能にする。

---

### 2-4. Undo/Redo の実装方式 [推奨]

| 選択肢 | 説明 | メリット | デメリット |
|--------|------|----------|-----------|
| **A. Command パターン** | 操作を Command オブジェクトとして記録 | 操作単位が明確、選択的 Undo 可能 | 実装量が多い |
| **B. スナップショット方式** | 状態全体のスナップショットを保存 | 実装がシンプル | メモリ消費が大きい |
| **C. Immer patches** | Immer の差分パッチを記録 | 差分のみ保存で効率的 | Immer 依存 |

**判断基準:**
- ボックス数が少ない（数十個）→ B で十分
- 大規模帳票（数百ボックス）→ A or C が望ましい

**推奨:** 想定されるボックス数の規模感に応じて決定する。

## Phase 3: 変数バインディングと型定義

### 3-1. 変数の記法（シンタックス） [必須]

テンプレート内で変数を示す記法。

| 選択肢 | 例 | メリット | デメリット |
|--------|-----|----------|-----------|
| **A. 二重波括弧** | `{{customerName}}` | 広く認知されている | Mustache/Handlebars と混同 |
| **B. ドル + 波括弧** | `${customerName}` | JS テンプレートリテラルと同一 | エスケープが必要な場面あり |
| **C. カスタム記法** | `%{customerName}` | 既存記法と衝突しない | 学習コスト |

**追加の判断:**
- ネスト（ドット記法）を許可するか？ 例: `{{customer.address.city}}`
- 配列アクセスを許可するか？ 例: `{{items[0].name}}`

**推奨:** 連携先システムのテンプレートエンジンの記法に合わせるのが望ましい。

---

### 3-2. 型定義の生成フォーマット [必須]

変数定義から生成する型定義の出力形式。

| 選択肢 | 出力例 | 用途 |
|--------|--------|------|
| **A. TypeScript interface** | `interface FormData { name: string; }` | TypeScript プロジェクト向け |
| **B. TypeScript type** | `type FormData = { name: string; }` | TypeScript プロジェクト向け |
| **C. JSON Schema** | `{ "type": "object", "properties": {...} }` | 言語非依存 |
| **D. A + C 両方** | TypeScript + JSON Schema を同時生成 | 汎用性最大 |

**推奨:** 連携先システムの技術スタックに依存する。TypeScript ベースなら A、多言語対応なら D。

---

### 3-3. 変数の型サポート範囲 [推奨]

変数に設定可能な型の範囲。

| 型 | 必要性 | 備考 |
|----|--------|------|
| `string` | 必須 | テキスト全般 |
| `number` | 必須 | 金額・数量 |
| `date` | 高 | 日付フォーマットとセットで検討 |
| `boolean` | 中 | チェックボックス的な用途 |
| `enum` | 中 | 選択肢リスト |
| `object` | 低 | ネスト構造（Phase 3 スコープ外でも可） |
| `array` | 低 | 繰り返し行（Phase 3 スコープ外でも可） |

## Phase 4: レイアウト微調整 UI

### 4-1. CSS 出力形式 [必須]

最終的なテンプレートの出力形式。連携先システムのインポート仕様に直結するため、早期に方向性を決めることが望ましい。

| 選択肢 | 説明 | メリット | デメリット |
|--------|------|----------|-----------|
| **A. HTML + CSS ファイル** | 静的ファイルとして出力 | シンプル、どこでも使える | フレームワーク非依存 |
| **B. React コンポーネント (TSX)** | React 用コンポーネントとして出力 | 型安全、変数バインディング容易 | React 依存 |
| **C. Web Component** | カスタム要素として出力 | フレームワーク非依存 | Shadow DOM の制約 |
| **D. JSON テンプレート定義** | レイアウト情報を JSON で出力し、ランタイムで描画 | 柔軟性最大 | 描画エンジンが別途必要 |

**推奨:** 連携先の「システムにインポートして運用する」仕様を確認した上で決定する。

---

### 4-2. カラーピッカーライブラリ [任意]

| 選択肢 | バンドルサイズ | 特徴 |
|--------|---------------|------|
| **react-colorful** | 約 2.8 KB (gzip) | 軽量、アクセシブル |
| **react-color** | 約 11 KB (gzip) | 多彩なプリセット（Sketch, Chrome 風） |
| **ネイティブ input[type=color]** | 0 KB | 依存なし、OS ネイティブ UI |

**推奨:** 初期は **ネイティブ input[type=color]** で十分。リッチな UI が必要になったら `react-colorful` を導入する。

---

### 4-3. フォント指定の範囲 [推奨]

出力テンプレートで使用可能なフォントの方針。

| 選択肢 | 説明 |
|--------|------|
| **A. システムフォントのみ** | OS 標準フォントに限定（ゴシック、明朝） |
| **B. Web フォント対応** | Google Fonts 等を利用可能にする |
| **C. カスタムフォント登録** | ユーザーがフォントファイルをアップロード |

**推奨:** 帳票用途では **A（システムフォント）** で十分なケースが多い。連携先の印刷環境に合わせて決定する。

## 横断的な判断事項

### X-1. 連携先システムのインポート仕様 [必須・早期確認推奨]

Phase 3（変数記法）と Phase 4（出力形式）の決定に直結する。以下の情報が早期にわかると、手戻りを防げる。

- テンプレートエンジンの種類（Mustache / Handlebars / 独自等）
- インポート時のファイル形式（HTML / JSON / 独自形式）
- 変数のバインディング方式
- 印刷エンジン（ブラウザ印刷 / PDF 生成ライブラリ等）

### X-2. 対応する Excel バージョン・機能範囲 [推奨・Phase 1 前]

変換対象とする Excel 機能の範囲。

| 機能 | 対応方針の候補 |
|------|---------------|
| セル結合 | 必須（帳票の基本構造） |
| 罫線スタイル | 必須（実線・破線・二重線等） |
| 条件付き書式 | 対応しない / 静的に評価 |
| 画像・図形 | 対応しない / 位置のみ抽出 |
| グラフ | 対応しない |
| マクロ (VBA) | 対応しない |
| 数式 | 対応しない / 計算結果のみ取得 |

**推奨:** 「80%自動・20%手動」の方針に基づき、帳票レイアウトに必要な最小限の機能に絞る。

### X-3. 想定されるボックス数の規模 [推奨・Phase 2 前]

パフォーマンス設計・Undo/Redo 方式の選定に影響する。

| 規模 | ボックス数 | 設計への影響 |
|------|-----------|-------------|
| 小規模 | 〜50 個 | 特別な最適化は不要 |
| 中規模 | 50〜200 個 | 仮想化の検討が必要 |
| 大規模 | 200 個〜 | Canvas 描画への切替検討 |
