# フェーズ別判断事項

> 最終更新: 2026-03-06

各フェーズの技術的判断事項と決定結果をまとめる。

---

## 凡例

| 記号 | 意味 |
|------|------|
| **決定済** | 方針が確定した項目 |
| **PoC 後決定** | PoC の結果を踏まえて最終決定する項目 |

---

## 前提条件

- 1ページあたり約 200 ボックス（中規模）
- 複数ページの読み込みを想定するが、同時に複数ページをレンダリングする必要はない

---

## Phase 1: Excel パーサー + プレビュー

### 1-1. Excel パースライブラリの選定 [PoC 後決定]

両ライブラリで PoC を組み（各1〜2時間）、制約 001〜004 で必要な情報がどちらの API で素直に取れるかを判断基準とする。結果は ADR-002 に記録する。

| 観点 | ExcelJS | SheetJS (xlsx) |
|------|---------|----------------|
| 設計思想 | Node.js 寄り（ストリーム・大ファイル対応） | ブラウザファースト（軽量） |
| バンドルサイズ | 約 1.5 MB | 約 800 KB（mini 版あり） |
| pageSetup 周りの型 | 明快（制約 003・004 の読み出しがやりやすい） | `!margins` / `!print` で取得可能 |
| ブラウザ相性 | 対応（ただし Node 向け設計） | ブラウザファースト |

**設計上の注意:** パースライブラリを使うのは Web 層（またはその境界）であり、ドメイン層には変換済みデータだけを渡す。ドメイン層の外部依存ゼロ制約は維持される。

---

### 1-2. プレビューのズーム・パン操作 [決定済]

**→ A. Fit-to-window のみ**

1ページずつの表示なら全体を画面に収めるだけで確認用途は成立する。ズーム・パンは Phase 2 以降で拡張する。

---

### 1-3. 複数ページの取り扱い [決定済]

**→ B. ページ選択式**

全ページをパースしてメモリに保持し、表示は選択ページのみ。同時レンダリングは不要。

---

## Phase 2: 調整エディタ

### 2-1. 状態管理ライブラリ [決定済]

**→ Zustand**

選定理由:
1. API がシンプルで導入コストが低い
2. ミドルウェアで Undo/Redo（Immer patches）を組みやすい
3. バンドルサイズが小さい（1.1 KB gzip）

中規模ボックス（〜200個）でも単一ストアで十分。Jotai の細粒度リアクティビティが活きるのは、もっとコンポーネント間の依存が複雑になった場合。

---

### 2-2. ドラッグ操作の実装方式 [決定済]

**→ A. ネイティブ SVG イベント**

SVG ベースのエディタで react-dnd 系を使うと、HTML DOM 前提のヒットテストと SVG 座標系の変換で余計な接合コードが増える。`mousedown → mousemove → mouseup` を自前で組むほうが制御しやすく、スナップ処理との統合もスムーズ。

---

### 2-3. スナップ・グリッドの粒度 [決定済]

| 項目 | 決定値 |
|------|--------|
| グリッド単位 | 1mm |
| スナップ距離 | 自動（ズーム連動） |
| ガイド線表示 | ドラッグ中のみ |

設定パネルでの変更機能は Phase 4 で追加する。

---

### 2-4. Undo/Redo の実装方式 [決定済]

**→ C. Immer patches**

選定理由:
- 中規模（〜200 ボックス）ではスナップショット方式はメモリ消費が大きい
- Command パターンは操作ごとにコマンドクラスを定義する実装コストが高い
- Immer patches なら差分のみ保存で効率的
- Zustand の immer ミドルウェアと自然に組み合わせられる

---

## Phase 3: 変数バインディングと型定義

### 3-1. 変数の記法 [決定済]

**→ A. 二重波括弧 `{{variableName}}`**

広く認知されていて迷いが少ない。連携先は自分たちのシステム。

| 機能 | 方針 |
|------|------|
| ドット記法（ネスト） | 許可する（例: `{{customer.address.city}}`） |
| 配列アクセス | Phase 3 では対象外。後から拡張する |

---

### 3-2. 型定義の生成フォーマット [決定済]

**→ A. TypeScript interface**

連携先も自分たちの TypeScript プロジェクトのため、まず interface で着手する。JSON Schema は必要になったら後追加。

---

### 3-3. 変数の型サポート範囲 [決定済]

| 型 | v1 対応 | 備考 |
|----|---------|------|
| `string` | 対応 | テキスト全般 |
| `number` | 対応 | 金額・数量 |
| `date` | 対応 | 日付フォーマットとセットで実装 |
| `boolean` | 後回し | 需要が出てから追加 |
| `enum` | 後回し | 需要が出てから追加 |
| `object` | Phase 3 スコープ外 | |
| `array` | Phase 3 スコープ外 | |

---

## Phase 4: レイアウト微調整 UI

### 4-1. 出力形式 [決定済]

**→ D. JSON テンプレート定義**

連携先が自分たちなら描画エンジン側を自由に作れるので JSON が最も柔軟。HTML 直出しだと後からフォーマットを変えたくなったとき全部作り直しになるが、JSON なら描画側だけ差し替えれば済む。

---

### 4-2. カラーピッカー [決定済]

**→ ネイティブ `input[type=color]`**

帳票用途のカラーは黒・グレー・白が大半。リッチなピッカーは過剰。必要になったら `react-colorful` を導入する。

---

### 4-3. フォント指定の範囲 [決定済]

**→ A. システムフォントのみ**

帳票は印刷前提のため、環境依存の Web フォントよりシステムフォント指定のほうが再現性が高い。

---

## 横断的な判断事項

### X-1. 連携先システムのインポート仕様 [決定済]

連携先は自分たちで開発するシステム。Phase 1〜2 の間に以下の仕様ドラフトを作成する。

| 項目 | 方針 |
|------|------|
| テンプレート形式 | JSON テンプレート定義 |
| 変数記法 | `{{variableName}}`（二重波括弧） |
| 型定義 | TypeScript interface |

Phase 3 着手前にドラフトを完成させることで、スムーズに開始できる。

---

### X-2. 対応する Excel 機能範囲 [決定済]

「80%自動・20%手動」の方針に基づき、帳票レイアウトに必要な最小限の機能に絞る。

| 機能 | 対応方針 |
|------|----------|
| セル結合 | **対応**（帳票の基本構造） |
| 罫線スタイル | **対応**（実線・破線・二重線等） |
| 条件付き書式 | 対応しない |
| 画像・図形 | 対応しない |
| グラフ | 対応しない |
| マクロ (VBA) | 対応しない |
| 数式 | 計算結果のみ取得 |

---

### X-3. 想定されるボックス数の規模 [決定済]

**→ 1ページあたり約 200 ボックス（中規模）**

| 設計への影響 | 対応方針 |
|-------------|----------|
| SVG レンダリング | Phase 1 では Fit-to-window で十分。性能問題が出たら仮想化を検討 |
| 状態管理 | Zustand 単一ストアで対応可能 |
| Undo/Redo | Immer patches（差分保存）で効率的に処理 |
| ドラッグ操作 | ネイティブ SVG イベントで制御可能 |
