# 設計原則

プロジェクト全体を貫く原則。Issue/PR の起票時に「この変更はどの原則に基づくか」を明示する。

## 基本原則

| # | 原則 | 要点 | 違反例 |
|---|------|------|--------|
| 1 | **責務の分離** | 一つの知識を複数モジュールが重複して持たない。データの種別（事実/状態/履歴/理由）ごとに所有者を明確にする | Web 層にボックス座標計算ロジックを直書きする |
| 2 | **参照はするが書き戻さない** | 他コンテキストのデータはスナップショット取得のみ。元システムへの書き戻し禁止 | Web 層から Domain 層の状態を直接変更する |
| 3 | **段階導入と独立動作** | 各モジュールは他モジュールが未導入でも単独で動作する。連携は「あれば便利」レベルに留める | エディタがないとパーサーの結果が確認できない |
| 4 | **ワンファクト・マルチビュー** | データは発生元で1度だけ記録する。各モジュールは同じデータを異なる視座で表示する | ボックス定義をプレビューとエディタで別々に保持する |
| 5 | **共通識別子による横断** | 全モジュール共通の識別子を設け、横断参照の基盤とする | ボックスIDの体系がプレビューとエディタで異なる |

## 並行開発ルール

複数の開発者・エージェントが同時に作業する際の制約。

| ルール | 内容 |
|--------|------|
| **パッケージ所有権** | 1パッケージを同時に2人以上が編集しない |
| **共通パッケージ変更時の義務** | 共通パッケージ（domain 等）を変更する場合、**変更前に** `docs/constraints/` に制約条件を記録する |
| **型の追加は自由** | 型の変更・削除は制約条件への記録が必須 |

## 段階導入の分解パターン

このプロジェクトは以下の順序で段階的に導入する:

```
Phase 1: Excel パーサー + プレビュー（パーサー単独で動作すること）
Phase 2: 調整エディタ（パーサー出力が安定してから着手）
Phase 3: 変数バインディング + 型定義（エディタ構造が固まってから）
Phase 4: レイアウト微調整 UI（上記が揃ってから拡張）
```

各段階で「この時点のアウトプットだけでも使えるか」を確認し、使えるなら次へ進む。

## コミットメッセージ規約

```
{type}({scope}): {summary}
```

| 要素 | 説明 | 例 |
|------|------|-----|
| `type` | 変更種別 | `feat`, `fix`, `docs`, `chore`, `refactor`, `test` |
| `scope` | 影響パッケージ | `domain`, `web`, `root` |
| `summary` | 変更内容の要約 | 日本語または英語 |

例:
```
feat(domain): ボックスモデルの型定義を追加
fix(web): SVGプレビューの座標計算を修正
docs(constraints): 001-box-model-coordinate-system を追加
chore(root): Biome 設定の更新
```

## エスカレーション基準

以下に該当する場合、`docs/issues/reports/` に事象を起票してユーザーに確認する:

| 基準 | 例 |
|------|------|
| **仕様書に記載がなく推測での実装が危険** | Excel罫線の種類マッピングが不明 |
| **設計原則が矛盾する状況** | ワンファクトと参照禁止が衝突する |
| **既に動作しているコードへの破壊的変更** | domain 型の削除・変更 |
| **Excel パースの精度に関わる判断** | 結合セルの分割アルゴリズムの選定 |
| **印刷精度に影響する変更** | mm座標系の丸め方式の変更 |
