# 事象管理

実装中に発生したブロッカーや課題を記録・追跡する。

## 目的

- ブロッカーの可視化と調査履歴の蓄積
- 同じ問題の再発防止
- エスカレーション基準の明確化

## 運用ルール

### 記録対象

- 実装の詰まり（技術的なブロッカー）
- 原因調査中の不具合
- 外部要因による作業停滞

### ファイル命名

```
docs/issues/reports/{連番3桁}-{kebab-case}.md
```

例: `001-excel-parse-merged-cell-offset.md`

### テンプレート

```markdown
# {連番}-{タイトル}

## 発生日
YYYY-MM-DD

## 対象レイヤー
domain / web / root

## 概要
（1〜2文で事象を要約）

## 発生状況
（何をしていたら発生したか）

## 調査経緯
（時系列で調査内容を記録）

- YYYY-MM-DD: 初期調査 — ...
- YYYY-MM-DD: 追加調査 — ...

## 関連ドキュメント
（関連する制約条件やADR）

## 解決方法
（解決した場合に記入）

## 状態
未解決 / 解決済

## 解決日
YYYY-MM-DD（解決した場合に記入）
```

### エスカレーション基準

以下に該当する場合、ユーザーに確認する:

| 基準 | 例 |
|------|------|
| 仕様書に記載がなく推測での実装が危険 | Excel罫線の種類マッピングが不明 |
| 設計原則が矛盾する状況 | ワンファクトと参照禁止が衝突する |
| 既に動作しているコードへの破壊的変更 | domain 型の削除・変更 |
| 外部ライブラリの制限に直面 | ExcelJS で取得できない情報がある |
