# 制約条件管理

実装過程で発見・決定された制約条件を記録する。

## 目的

- 設計判断のトレーサビリティを確保する
- 後から参加する開発者・エージェントが「なぜこうなっているか」を理解できるようにする
- 制約の解消時期と方法を管理する

## 運用ルール

### 記録対象

- 暫定措置（後で解消が必要な技術的制約）
- 設計判断により生じた制約（トレードオフの選択）
- レイヤー間の結合に関する取り決め
- 外部依存・インフラに起因する制限

### ファイル命名

```
docs/constraints/{連番3桁}-{kebab-case}.md
```

例: `001-box-model-coordinate-system.md`

### テンプレート

```markdown
# {連番}-{タイトル}

## 発生日
YYYY-MM-DD

## 対象レイヤー
domain / web / root

## 概要
（1〜2文で制約の内容を要約）

## 背景
（なぜこの制約が生じたか）

## 詳細
（制約の具体的な内容）

## 影響範囲
（この制約によって影響を受けるモジュール・機能）

## 今後の対応
（解消予定の有無、解消方法の見込み）

## 状態
有効 / 解消
```

### 記録パターン

| パターン | 例 |
|---------|-----|
| 骨格先行 → 仕様後追い | 型定義を先に作り、実装詳細は後から確定 |
| 責務境界の決定 | Excel パースと座標変換の責務分離を決定 |
| 外部依存の制約 | ExcelJS の API 制限に起因するパース仕様 |
| 精度の線引き | mm 座標への変換で小数第2位以下を切り捨て |

### 制約解消時の手順

1. ドキュメント内に「解消日」と「解消方法」を追記
2. 一覧テーブルの状態を「解消」に変更
