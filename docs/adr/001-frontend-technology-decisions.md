# ADR-001: フロントエンド技術選定

- 状態: 提案（ユーザー確認待ち）
- 日付: 2026-03-05
- 対象: バックログ 001 Step A（UI シェル）

## コンテキスト

UI シェル実装にあたり、以下の技術選定が必要。いずれも新規ライブラリの導入有無に関わる判断であり、後続フェーズへの影響が大きい。

---

## 決定事項

### 1. CSS 手法: CSS Modules

**選択:** `.module.css` ファイル（Vite 標準サポート）

**代替案:**
- Plain CSS — グローバルスコープ汚染のリスク
- CSS-in-JS（styled-components, Emotion）— 新規依存追加、ランタイムコスト
- Tailwind CSS — 新規依存追加、ユーティリティクラスの学習コスト

**理由:**
- Vite が設定不要でサポート
- スコープ付きクラス名でコンポーネント間の衝突を防止
- lint-staged が `.css` ファイルを既にカバー
- 最小限の複雑さで十分な機能

**ユーザー判断ポイント:** Tailwind CSS の導入を希望する場合は早期に決定が必要。CSS Modules からの移行は段階的に可能。

---

### 2. 状態管理: React 組み込み（useState + useReducer）

**選択:** ライブラリ追加なし。React 標準フックのみ

**代替案:**
- Zustand — 軽量で人気。ただし現時点では不要
- Jotai — アトミック状態管理。エディタ機能で有用かもしれない
- Redux Toolkit — オーバーエンジニアリング

**理由:**
- Step A の状態は最小限（アップロードファイル、デフォルト用紙設定）
- クロスコンポーネント状態が不要
- Phase 2（エディタ機能）で状態が複雑化した段階で再評価

**ユーザー判断ポイント:** エディタフェーズ（ボックス選択、undo/redo、プロパティ編集）で状態管理ライブラリが必要になる見込み。Zustand vs Jotai の判断はその時点で。

---

### 3. モックプレビュー: 空の用紙アウトライン

**選択:** 実在の `paper` ドメイン型を使用した A4 用紙アウトライン + 印刷可能領域（破線）

**代替案:**
- ダミーボックス表示 — box/line ドメイン型が未実装のため、架空のデータ構造を作ることになる
- 完全に空の表示 — SVG 座標系の検証ができない

**理由:**
- 既存の `createPaperDefinition` + `getPaperDimensions` を使用し、ドメイン連携パスを検証
- SVG `viewBox` を mm 単位で設定し、Step B でボックス座標をそのまま描画可能に
- 架空のモックデータ形式を作らない（ドメイン型との乖離リスク回避）

---

### 4. カラースキーム: ニュートラルグレー / ホワイト

**選択:** ツール指向の機能的 UI（Figma / VS Code ライク）

| 要素 | 色 |
|------|-----|
| ヘッダー背景 | `#2d2d2d` |
| ヘッダー文字 | `#ffffff` |
| サイドバー背景 | `#f5f5f5` |
| キャンバス背景 | `#e0e0e0` |
| 用紙 | `#ffffff` |
| アクセント | `#2563eb` |
| ボーダー | `#d1d5db` |

**ユーザー判断ポイント:** ダークモード対応を将来実装する場合、CSS カスタムプロパティ（現在採用済み）を活用して切り替え可能。現時点ではライトテーマのみ。

---

## 影響範囲

- `src/web/` 全体の CSS 記述方式
- 将来のコンポーネント追加時のスタイリングパターン
- Phase 2 での状態管理ライブラリ選定
